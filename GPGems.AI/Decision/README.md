# 融合决策系统

> 多层 AI 决策系统的现代化 C# 实现，模拟人类日常决策行为

---

## 🎯 架构总览

本系统采用 **三层互补架构**，每层解决一类特定问题，通过行为树协同工作。


## 📋 设计总览

### 融合架构（3层结构）

```
┌─────────────────────────────────────────────────────────┐
│  顶层：FSM 状态机              【流程控制】                │

│  └───┬────┘  └───┬────┘  └───┬────┘  └───┬────┘          │
└──────┼────────────┼────────────┼────────────┼────────────┘
       │            │            │            │
       ▼            ▼            ▼            ▼
┌─────────────────────────────────────────────────────────┐
│  中层：行为树（每状态一棵）    【行为编排】                │
│  ┌───────────────────────────────────────────────────┐  │
│  │  上班行为树                                        │  │
│  │   Sequence:                                       │  │
```

## 🎯 各层职责边界

| 层级 | 技术 | 职责 | 管什么 | 不管什么 |
|------|------|------|--------|----------|
| **顶层 FSM** | 状态机 | 流程控制 | 什么时候该"上班"还是"下班" | 上班具体做什么 |
| **中层 BehaviorTree** | 行为树 | 行为编排 | 上班先做A再做B还是并行做 | A/B如何选择 |
| **叶子节点** | 模糊逻辑 | 连续值判断 | 工作压力"大不大"、精力"够不够" | 选哪个任务 |
| **叶子节点** | 效用系统 | 动作评分 | 写代码 vs 开会 vs 摸鱼 哪个分数高 | 动作如何执行 |
| **叶子节点** | GOAP | 路径规划 | 完成任务需要的步骤序列 | 步骤该不该做 |
| **全局唯一** | 黑板 | 数据共享 | 所有中间结果、全局状态 | 任何决策逻辑 |

## 📋 各层定位详解

### 1. FSM 有限状态机
**📍 位置**: `StateMachine/`

| 属性 | 说明 |
|------|------|
| **职责** | 大场景时间驱动切换 |
| **粒度** | 最粗 - 5 个状态 |
| **触发** | 时间、日期、外部事件 |
| **必需性** | ✅ 核心框架 |

**状态列表**:
- `Morning` 🌅 早上 (起床 ~ 通勤)
- `Working` 💼 工作中 (上班时间)
- `Evening` 🌆 晚间 (下班 ~ 睡前)
- `Sleeping` 😴 睡觉中 (夜间)
- `Weekend` 🎉 周末 (休息日)

**设计原则**:
> FSM 只负责"现在是什么场景"，不负责"场景内做什么"。
> 内部通过 `_currentPhase` 字符串跟踪阶段，不使用子状态机。

---

### 2. BehaviorTree 行为树
**📍 位置**: `BehaviorTree/`

| 属性 | 说明 |
|------|------|
| **职责** | 子系统调度、行为组织、条件分发 |
| **粒度** | 中等 - 节点级 |
| **触发** | FSM 状态切换时加载对应行为树 |
| **必需性** | ✅ 集成胶水 |

**主要节点类型**:
- `Selector` 选择节点（一成功则成功）
- `Sequence` 序列节点（全成功才成功）
- `Condition` 条件节点
- `Action` 动作节点

**设计原则**:
> 行为树是"交换机"，不是"决策者"。它把决策委托给下层的专用系统。

---

### 3. FuzzyLogic 模糊逻辑
**📍 位置**: `FuzzyLogic/`

| 属性 | 说明 |
|------|------|
| **职责** | 模糊输入 → 量化输出 |
| **粒度** | 变量级 |
| **触发** | 每帧计算，输出到黑板 |
| **必需性** | ✅ 人类决策不是 0/1 |

**内置引擎**:
- **MoodEngine** 心情引擎: 精力 + 压力 + 疲劳 → 心情值
- **WorkEngine** 工作引擎: 精力 + 压力 → 工作强度

**模糊集合示例**:
```
精力: 极低 ← 低 ← 中等 ← 高 → 极高
      [0, 20] [15, 40] [30, 70] [60, 85] [80, 100]
```

**设计原则**:
> 人类不会因为精力=50 就突然从"认真"变"摸鱼"。
> 模糊逻辑让决策有平滑的过渡区间。

---

### 4. UtilitySystem 效用系统 (NeedBased)
**📍 位置**: `UtilitySystem/`

| 属性 | 说明 |
|------|------|
| **职责** | 多行动竞争选择 |
| **粒度** | 单步行动 |
| **触发** | 每帧选择得分最高的行动 |
| **必需性** | ✅ 日常决策核心 |

**这就是 NeedBased**:
效用系统本质就是需求驱动决策。每个行动的得分 = 这个行动能多大程度满足当前最迫切的需求。

**内置行动 (12 个)**:
| 类别 | 行动 | 满足的需求 |
|------|------|-----------|
| 📝 工作 | `writeCode` `slackOff` | 成就感 / 摸鱼 |
| 😌 休息 | `takeBreak` `goToilet` | 精力恢复 |
| 🍽️ 生理 | `eatMeal` `drinkWater` | 饥饿 / 口渴 |
| 🎮 休闲 | `playGame` `watchVideo` `socialMedia` `readBook` | 娱乐 |
| 💬 社交 | `chat` | 社交需求 |
| 🧹 卫生 | `shower` | 清洁需求 |

**考量曲线**:
- `LinearCurve` 线性曲线
- `StepCurve` 阶梯曲线
- `GaussianCurve` 高斯曲线

**设计原则**:
> 没有"正确"的行动，只有"当前最合适"的行动。
> 每个行动都有机会被选中，只是概率不同。

---

### 5. GOAP 目标导向行动规划
**📍 位置**: `Goap/`

| 属性 | 说明 |
|------|------|
| **职责** | 多步骤任务规划 |
| **粒度** | 行动序列 |
| **触发** | 仅在有明确目标时激活 |
| **必需性** | ⚠️ 可选但有价值 |

**与 UtilitySystem 的本质区别**:

| 维度 | 效用系统 | GOAP |
|------|---------|------|
| 决策单位 | 单步行动 | 多步行动序列 |
| 前瞻能力 | 只看当前 | 看未来 N 步 |
| 行动关系 | 独立 | 有依赖链 |
| 适用场景 | 日常需求满足 | 复杂任务完成 |

**工作流程**:
```
1. 目标出现 (如: "提交报告")
   ↓
2. 规划器搜索状态空间
   收集资料 → 分析数据 → 撰写报告 → 提交
   ↓
3. 按序列执行每一步
   ↓
4. 完成后回到效用系统
```

**设计原则**:
> GOAP 不是每帧必跑的核心路径，而是按需激活的增强功能。
> 日常作息 99% 的时间不需要规划器。

---

### 6. Blackboard 黑板
**📍 位置**: `Blackboard/`

| 属性 | 说明 |
|------|------|
| **职责** | 全局状态共享、事件通知 |
| **粒度** | 键值对 |
| **触发** | 读写操作、过期清理 |
| **必需性** | ✅ 数据总线 |

**核心特性**:
- 线程安全读写
- TTL 自动过期
- Observer 观察者模式
- 类型安全的泛型 API

**常用键**:
```
hour_of_day      float   当前时间 (0-24)
energy           float   精力 (0-100)
hunger           float   饥饿 (0-100)
stress           float   压力 (0-100)
fatigue          float   疲劳 (0-100)
work_intensity   float   工作强度 (0-1)
boss_nearby      bool    老板是否在场
is_weekend       bool    是否周末
```

## 🔗 子系统协作流程

以"工作状态"为例：

```
[9:00 时间到]
    ↓
FSM: Morning → Working
    ↓
加载 WorkBehavior 行为树
    ↓
┌─ 模糊逻辑每帧计算:
│   精力(70) + 压力(20) → 工作强度(0.8)
│
├─ 效用系统每帧选择:
│   writeCode(0.85) > slackOff(0.3) > takeBreak(0.2)
│   → 执行写代码
│
└─ GOAP 待机 (无目标)


[12:00 饥饿升到 80]
    ↓
效用系统重计算:
   eatMeal(0.95) > writeCode(0.7) > ...
   → 执行吃饭
    ↓
饥饿下降，继续工作


[出现"提交报告"任务]
    ↓
GOAP 激活，规划序列:
   收集资料 → 分析数据 → 撰写报告 → 提交
    ↓
按步骤执行，完成后回到效用系统
```

## 🧠 为什么是五种决策算法而不是一种？

| 如果只用... | 问题 |
|------------|------|
| **纯 FSM** | 状态爆炸，每个动作组合都要新状态 |
| **纯行为树** | 硬编码逻辑，缺乏动态权衡 |
| **纯效用** | 只能单步决策，无法处理多步任务 |
| **纯 GOAP** | 规划开销大，简单日常没必要 |
| **纯模糊** | 只有计算，没有决策框架 |

**五种互补** = 日常决策的流畅性 + 复杂任务的前瞻性

## 🎮 扩展指南

**添加新行动**:
1. 在效用系统加 `UtilityAction`
2. 定义考量曲线和评分逻辑

**添加新状态**:
1. 在 FSM 加新 `DelegateState`
2. 写对应的行为树

**添加新任务**:
1. 定义 `GoapAction`（前置条件 + 效果）
2. 定义 `GoapGoal`
3. 规划器自动寻找路径
