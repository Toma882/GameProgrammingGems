## 基础搜索算法

### BFS 广度优先搜索

**核心思想**：按层级逐层扩展，像水波一样扩散出去。

```
      ★ → ● → ● → ●
      ↓   ↓   ↓   ↓
      ● → ● → ● → ●
      ↓   ↓   ↓   ↓
      ● → ● → ● → ◎
```

**实现细节**：
- **数据结构**：`Queue<T>` 先进先出
- **时间复杂度**：O(V + E)
- **空间复杂度**：O(V)

**特点**：
- ✅ 在单位权重网格中保证找到最短路径
- ✅ 实现简单，直观易懂
- ❌ 搜索范围大，效率低
- ❌ 不支持非单位权重

**适用场景**：
- 简单迷宫游戏
- 状态空间搜索
- 最短路径步数计数

---

### DFS 深度优先搜索

**核心思想**：一条路走到黑，碰壁才回头。

```
      ★ → → → →
                ↓
      ← ← ← ← ←
      ↓
      → → → → ◎
```

**实现细节**：
- **数据结构**：`Stack<T>` 后进先出
- **时间复杂度**：O(V + E)
- **空间复杂度**：O(V)

**特点**：
- ✅ 内存占用小（栈深度）
- ✅ 实现极其简单
- ❌ 路径可能非常长
- ❌ 不保证最短路径
- ❌ 可能陷入死胡同

**适用场景**：
- 教学演示
- 状态空间遍历
- 记忆化搜索

---

### Dijkstra 算法

**核心思想**：始终扩展当前代价最小的节点，动态规划思想。

```
      ★(0) → ●(1) → ●(2)
        ↘     ↓     ↙
          ●(1.4) → ●(2.8) → ◎
```

**实现细节**：
- **数据结构**：优先队列 `PriorityQueue`
- **时间复杂度**：O((V + E) log V)
- **核心公式**：`g(neighbor) = g(current) + cost(current, neighbor)`

**特点**：
- ✅ 保证找到最短路径（全局最优）
- ✅ 支持任意非负权重
- ❌ 搜索方向盲目，向所有方向扩展
- ❌ 对于长距离寻路效率很低

**适用场景**：
- 路由协议（OSPF）
- 交通导航
- 加权图最短路径

---

### Best-First 贪心搜索

**核心思想**：只看启发式，始终向最接近目标的方向前进。

```
      ★
        ↘
          ●
            ↘
              ● → ◎
```

**实现细节**：
- **数据结构**：优先队列
- **排序键**：仅启发式距离 `h(n)`
- **时间复杂度**：O((V + E) log V)

**特点**：
- ✅ 速度极快，搜索范围小
- ✅ 很少绕路
- ❌ 不保证最短路径
- ❌ 可能陷入局部最优

**适用场景**：
- 快速原型验证
- 对路径质量要求不高的场景
- 实时性要求高的游戏

## A* 算法详解

### 核心原理

A* 是 Dijkstra 和 Best-First 的完美结合：

**f(n) = g(n) + h(n)**

- **g(n)**：从起点到当前节点的实际代价
- **h(n)**：当前节点到终点的启发式估计代价
- **f(n)**：总代价估计

**可采纳性**：当启发式不高估实际代价时，A* 保证找到最短路径

---

### 启发式函数

#### 1. 曼哈顿距离（Manhattan）
```csharp
h = |x1 - x2| + |y1 - y2|
```
- ✅ 4方向网格的完美可采纳启发式
- ❌ 8方向网格会高估

#### 2. 切比雪夫距离（Chebyshev）
```csharp
h = max(|x1 - x2|, |y1 - y2|)
```
- ✅ 对角线移动代价 = 直线移动代价时使用
- ❌ 代价不同时会高估

#### 3. Octile 距离（8方向网格最优）
```csharp
int dx = Abs(x1 - x2);
int dy = Abs(y1 - y2);
h = min(dx, dy) * 1.41421356 + Abs(dx - dy);
```
- ✅ 8方向网格的完美可采纳启发式
- ✅ 对角线代价 √2，直线代价 1
- ✅ 当前实现使用

#### 4. 欧几里得距离
```csharp
h = sqrt(dx² + dy²)
```
- ✅ 任意方向移动时使用
- ❌ 计算开销大（浮点数运算）

---

### A* 优化手段

#### ✅ 优化 1：使用 .NET 内置 PriorityQueue

**问题**：原来使用 `SortedSet` 需要自定义比较器，性能较低

**优化后**：
```csharp
// 优化前
var openSet = new SortedSet<(double f, int x, int y)>();

// 优化后
var openSet = new PriorityQueue<GridNode, double>();
openSet.Enqueue(node, node.FCost);
```

**性能提升**：
- 插入：O(log n) → 相同复杂度但常数更小
- 提取最小值：O(log n) → 原生支持
- 内存开销减少约 30%

---

#### ✅ 优化 2：Jump Point Search (JPS)

**核心思想**：跳过对称路径上的大部分节点，只在"跳点"处展开

```
普通 A*:  ● → ● → ● → ● → ●
JPS:      ● → → → → → → → ●
```

**优化效果**：
- 速度提升 10-100 倍
- 保持最优性
- 开放集大小减少 90%+

**关键规则**：
1. 直线移动：跳过强制邻居
2. 对角线移动：只在有强制邻居时停止
3. 跳点检测

---

#### ✅ 优化 3：Theta* 任意角度寻路

**问题**：网格路径有锯齿效应

```
网格路径:    ～～
            ～
         ～
      ～

Theta*:  ────────
```

**优化方法**：
- 在重建路径时检查视线
- 父节点不必是相邻节点
- 路径平滑自然

---

#### ✅ 优化 4：层级 A* (HPA*)

**核心思想**：
1. 将大图分块（cluster）
2. 先在块级粗粒度寻路
3. 再在块内细粒度寻路

**适用**：
- 大型开放世界游戏
- 万级节点以上的超大地图

---

#### ✅ 优化 5：Early Exit 提前终止

```csharp
// 传统：找到目标立即返回
if (current == goal)
    return ReconstructPath(start, goal);
```

**优化**：
- 不必等待所有可能路径都检查
- 找到第一个就是最优（启发式可采纳前提下）

---

#### ⚠️ 优化 6：邻居排序（有条件优化！）

```csharp
// 按启发式距离排序邻居，优先扩展更有希望的方向
var neighbors = map.GetNeighbors(current)
    .OrderBy(n => OctileDistance(n, goal))
    .ToList();
```

**核心原理**：
> **用 CPU 排序时间，换取更少的节点扩展**

**实测效果（强烈依赖地图类型）**：

| 地图类型 | 效果 | 说明 |
|---------|------|------|
| **空旷地图** | ❌ **反向优化，慢 60-90%** | 没有岔路，排序 0 收益，纯开销 |
| 稀疏障碍物 | ⚖️ 差不多 | 收益勉强覆盖开销 |
| **密集迷宫** | ✅ **快 20-40%** | 少走很多死胡同，省下的时间 > 排序开销 |

**建议**：
- 默认关闭，只在确认是迷宫场景时启用
- 或者运行时动态检测障碍物密度自动开关

---

#### ⚠️ 优化 7：路径平滑（后处理，可选）

```
原始路径:   ● → ●
              ↘
                ● → ●
                    ↘
                      ●

平滑后:     ● ──────────── ●
```

**算法**：Bresenham 视线检测，O(n²) 复杂度

**实测效果**：

| 地图类型 | 效果 | 说明 |
|---------|------|------|
| **空旷地图** | ❌ **拖慢 40%** | 直线不需要平滑，纯计算开销 |
| 有障碍物 | ⚖️ 路径好看但慢 | 视觉效果提升，代价是额外时间 |

**建议**：
- **默认关闭**，作为后处理选项
- 只有需要展示路径给玩家看时才开启
- AI 内部寻路完全不需要（节点列表够用了）

---

#### ✅ 优化 8：动态权重 A*

```csharp
// 动态调整启发式权重
double weight = IsFarFromGoal ? 1.5 : 1.0;
f = g + weight * h;
```

**效果**：
- 远距离时更贪心，速度更快
- 接近目标时恢复最优性

## 双向 A* 算法

### 核心原理

同时从起点和终点两个方向搜索，在中间相遇：

```
正向:  ★ → ● → ● ← ● ← ◎
                ↓
                相遇点
```

### 实现细节

**数据结构**：
- 两个优先队列：`forwardOpen` + `backwardOpen`
- 两个闭集：`forwardClosed` + `backwardClosed`
- 两个 G 值字典

### 相遇检测

```csharp
// 当前节点在对方闭集中时，计算总代价
if (backwardClosed.Contains(current))
{
    double totalCost = forwardG[current] + backwardG[current];
    if (totalCost < bestCost)
    {
        bestCost = totalCost;
        meetingPoint = current;
    }
}
```

### 路径拼接

```csharp
// 前向路径: start → ... → meeting
// 反向路径: meeting → ... → goal
// 合并:     start → ... → meeting → ... → goal
```

### 性能优势

| 距离 | 单向 A* | 双向 A* | 提升 |
|------|---------|---------|------|
| 10 步 | 100 节点 | ~30 节点 | 3.3x |
| 100 步 | 10,000 节点 | ~300 节点 | 33x |
| 1000 步 | 1,000,000 节点 | ~3,000 节点 | 333x |

**搜索区域**：O(r²) → O(2 × (r/2)²) = O(r²/2)
**实际提升**：长距离 50-70% 速度提升

## 总结

### 选择建议（游戏开发版）

| 游戏场景 | 推荐算法 | 为什么 |
|---------|---------|--------|
| **玩家角色** | 经典 A* | 稳定、最优、不会出问题 |
| **RTS 大部队跨地图** | 双向 A* | 长距离性能提升 50%+ |
| **MMO 服务器寻路** | 并行双向 A* | 多核 CPU 发挥最大性能 |
| **小兵/野怪巡逻** | Best-First | 极致速度，绕路玩家也看不出来 |
| **地牢迷宫/复杂关卡** | A* 优化版 | 障碍物密集时邻居排序有正收益 |
| **网格解谜游戏** | BFS | 保证步数最少，实现简单 |
| **地形成本不同（草地/山地）** | Dijkstra / 加权 A* | 唯一支持动态权重的最优算法 |
| **开放世界大地图** | HPA* 分层寻路 | 先分块粗查，再细查 |

### 一句话速查

| 算法 | 什么时候用 | 一句话 |
|------|-----------|--------|
| **BFS** | 简单 2D 网格游戏 | 最笨但最稳的算法 |
| **DFS** | 生成迷宫、教学 | 一条路走到黑 |
| **Dijkstra** | 地形有不同代价 | 全局最优，但最慢 |
| **Best-First** | 非关键单位 | 最快，但可能绕路 |
| **经典 A*** | 90% 的游戏场景 | 速度和质量的完美平衡 |
| **双向 A*** | 地图 > 100×100 | 长距离寻路神器 |
| **并行双向 A*** | 服务器端、多核 | 榨干 CPU 最后一点性能 |
| **优化版 A*** | 迷宫多、路径要展示 | 好看但慢，按需开启 |

### 关键洞察

1. **启发式质量决定一切**：好的启发式 = 快 + 准
2. 双向搜索在长距离场景下收益巨大
3. 数据结构选择影响性能 30%+
4. 后处理平滑让路径看起来更自然
5. 没有免费午餐：速度 ↔ 质量的权衡永恒

## 高级寻路与群体算法

### 算法矩阵

| 算法 | 类型 | 适用单位数 | 性能 | **核心优势** | **典型场景** |
|------|------|-----------|------|-------------|-------------|
| **A*** | 单体寻路 | 1个 | O(log n) | 精准、最优路径 | 玩家角色、英雄单位 |
| **双向A*** | 单体寻路 | 1个 | O(log n) | 长距离提速50%+ | 跨地图移动、RTS |
| **分层寻路** | 单体 | 1个 | O(log n) | 大地图提速5-20倍 | 开放世界、MMO |
| **流场寻路** | 群体 | 50-1000+ | **O(1) per单位** | 百级单位共享路径 | 人群、小兵、丧尸 |
| **ORCA避障** | 局部 | N个 | O(N²) | 无穿透、无抖动 | 战斗单位移动 |
| **社会力模型** | 群体 | N个 | O(N²) | 模拟真实人群行为 | 排队、疏散、社交 |
| **ORCA + 流场** | 组合 | 200+ | O(N) | 大规模平滑移动 | 超大规模RTS |

---

### 流场寻路 FlowField

**核心原理**：一次计算，所有单位共享

```
目标点 ◎
  ↓
生成积分场（Cost Field）
  ↓  
计算每个格子指向目标的向量（Flow Field）
  ↓
所有单位查询当前格子的向量，直接移动
```

**性能对比**：

| 单位数 | A*总时间 | 流场总时间 | 提升 |
|-------|---------|-----------|------|
| 1 | 0.6ms | 2ms | **A*更快** |
| 10 | 6ms | 2.1ms | 3x |
| 50 | 30ms | 2.5ms | 12x |
| 100 | 60ms | 3ms | **20x** |
| 500 | 300ms | 5ms | **60x** |

**适用**：
✅ 大量单位同目标（RTS大部队）
✅ 人群模拟、丧尸潮
✅ 紧急疏散模拟
❌ 每个单位目标不同
❌ 目标频繁变化

---

### ORCA 互惠碰撞避免

**核心问题**：传统"各让各的"会出现死锁

```
单位A →  ← 单位B

传统避障：A右让，B也右让 → 撞在一起！
ORCA：双方各承担50%避让责任 → 平滑通过
```

**ORCA vs 传统分离**：

| 特性 | 传统Steering | ORCA |
|------|-------------|------|
| 碰撞穿透 | 频繁 | 几乎不会 |
| 抖动 | 严重 | 极少 |
| 人群密度1人/m² | 卡顿 | 流畅 |
| 人群密度2人/m² | 卡住 | 基本正常 |
| 瓶颈通过效率 | 低 | 高 |

**典型代码**：
```csharp
var orca = new ORCASimulation();
orca.AddAgent(new Vector2(0, 0), 0.5f, 3f);
orca.AddAgent(new Vector2(5, 0), 0.5f, 3f);
orca.SetTargetForAll(new Vector2(2.5f, 0));  // 对头碰
orca.Update(0.05f);  // ORCA会自动让双方各让一侧
```

### 社会力模型 Social Force

**用物理学模拟人类行为**：

```
驱动力：我想到目标点 → F = m * (v_desired - v_current) / τ
      ↓
排斥力：离别人远点 → F = A * exp(-distance / B)
      ↓
吸引力：跟朋友近点 → 社交团体力
      ↓
边界力：别撞墙 → 障碍物排斥
```

**行为调参**：

| 参数 | 效果 | 典型值 |
|------|------|--------|
| A | 排斥力强度 | 1-3 |
| B | 排斥力作用范围 | 0.2-0.5 |
| τ | 反应时间 | 0.3-0.8 |
| 半径 | 个人空间 | 0.3-0.6m |

**可调行为**：
- 👫 朋友走在一起（增加吸引力）
- 🚶 陌生人保持距离（增大排斥半径）
- 🏃 紧急疏散（增大期望速度）
- 🚧 排队行为（前方吸引力 + 侧方排斥）

### 分层寻路 Hierarchical Pathfinding

**核心思想**：抽象 + 缓存

```
Step 1: Chunk层（宏观）
  把地图分成10x10大块
  只看块之间有没有通道
  A*搜索：原图10000节点 → 抽象图100节点 ✅

Step 2: Cell层（微观）
  只在经过的Chunk内精细寻路
  搜索空间减少90%+
```

**性能提升**：

| 地图大小 | 普通A* | 分层A* | 提升 |
|---------|--------|--------|------|
| 100x100 | 0.6ms | 0.2ms | 3x |
| 500x500 | 15ms | 2ms | 7.5x |
| 2000x2000 | 200ms | 10ms | **20x** |

### 推荐组合拳

#### 🏰 中大型经营游戏（100人以内）
```
流场寻路 + ORCA局部避障
    ↓
宏观路径：流场一次计算，百人共享
微观调整：ORCA处理人与人之间不穿透
```

#### 🏙️ 超大城市/战场（千人级）
```
分层寻路（宏观） + 流场（中观） + ORCA（微观）
    ↓
跨城区：分层寻路找抽象路径
街区内：流场指导大方向
人与人：ORCA防穿透
```

#### 🏪 商店/排队模拟
```
社会力模型 + 吸引力场
    ↓
排队：目标点吸引力 + 前方排斥（保持距离）
人流：与墙壁保持距离 + 同向者吸引力
```
