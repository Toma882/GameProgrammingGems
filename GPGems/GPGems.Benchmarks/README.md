# 寻路算法基准测试

使用 BenchmarkDotNet 对所有 8 种寻路算法进行专业的性能对比。

## 运行测试

```bash
# 进入项目目录
cd GPGems.Benchmarks

# 运行完整基准测试（Release 模式）
dotnet run -c Release
```

## 测试参数

| 参数 | 取值 | 说明 |
|------|------|------|
| `GridSize` | 25, 50, 100 | 不同网格大小测试算法的可扩展性 |
| `ObstacleDensity` | 0.0, 0.2 | 空旷地图 vs 20% 障碍物密度 |

## 结果解读

基准测试输出的关键列：

| 列名 | 说明 |
|------|------|
| `Mean` | 多次运行的**平均耗时**（最可信指标） |
| `Error` | 统计误差（99.9% 置信区间） |
| `StdDev` | 标准差（数据稳定性，越小越好） |
| `Ratio` | 相对于基准算法（A*）的性能倍数 |
| `Rank` | 性能排名（A 最快，E 最慢） |
| `Gen 0/1/2` | 各代 GC 回收次数（越小越好） |
| `Allocated` | 单次运行分配的内存大小 |

## 快速测试

完整测试需要 10-30 分钟。如果只想快速验证，可以修改 `PathfindingBenchmarks.cs`：

```csharp
[Params(25)]  // 只测试小地图
public int GridSize { get; set; }

[Params(0.0)] // 只测空旷地图
public double ObstacleDensity { get; set; }
```

## 预期结论（典型情况）

| 算法 | 小地图表现 | 大地图表现 | 内存占用 |
|------|-----------|-----------|---------|
| Best-First | 最快 | 非常快 | 低 |
| 经典 A* | 优秀 | 优秀 | 中等 |
| 双向 A* | 良好 | **优秀** | 中等 |
| 并行双向 A* | 可能稍慢（线程开销） | **优势明显** | 高 |
| BFS | 中等 | 较差 | 高 |
| Dijkstra | 中等 | 较差 | 高 |
| DFS | 不稳定 | 不稳定 | 低 |

---

## ⚠️ A* 优化版 场景分析

**实测结论：优化版在空旷地图上是反向优化，慢 60-90%！**

| 地图类型 | 优化版效果 | 原因 |
|---------|-----------|------|
| **空旷地图** | ❌ **慢 60-90%** | 排序开销 > 收益 |
| 稀疏障碍物 | ⚖️ 差不多 | 收益勉强覆盖开销 |
| **密集迷宫** | ✅ 快 20-40% | 少走很多死胡同 |

### 三大"优化"的真实代价

| 优化手段 | 空旷地图 | 迷宫 | 建议 |
|---------|---------|------|------|
| 邻居 `.OrderBy()` | 纯开销，0 收益 | 少扩展 20-40% 节点 | 迷宫才开 |
| 路径平滑 + Bresenham | O(n²) 开销 | 有用（路径更短） | 默认关闭，可选 |
| 动态权重 | 多余计算 | 加快早期搜索 | 视需求 |

### 为什么会这样？

```
邻居排序 = 用 CPU 时间 换 OpenSet 质量

空旷地图：没有岔路 → 排序后 OpenSet 质量不变 → 纯亏
迷宫地图：岔路很多 → 排序后少走死胡同 → 省下的时间 > 排序开销
```

> 💡 真实游戏中的地图通常障碍物都不少，优化版在实际场景中可能还是正收益。建议用你自己的游戏地图数据测试！

---

> **注意**：多线程算法在小地图上可能因为线程调度开销反而比单线程慢，但在大地图长距离寻路时优势会显现。
